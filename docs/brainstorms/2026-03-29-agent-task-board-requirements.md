# Agent Task Board

## Problem Frame
本项目要提供一个面向本地单人使用的 agent 任务看板，让人类可以把代码任务发布给 `Claude Code` 或 `Codex CLI` 执行，并在统一的 CLI 与桌面 UI 中观察任务生命周期、执行进度、验证结果与重新派发过程。

当前痛点不是“缺一个待办列表”，而是缺一套稳定的本地调度与观测能力：当前工作目录下的任务如何被派发给 agent、如何选择 agent、何时自动启动、如何回传日志、失败后如何重开、验证不通过后如何再次进入执行链路。MVP 的目标是先把这条单任务单 agent 的闭环做通，而不是一开始做成团队级调度平台或通用工作流引擎。

## Requirements

**任务定义**
- R1. 人类可以创建任务，并填写至少这些字段：标题、任务说明、本次执行的目标 agent、流程。
- R2. 系统必须以当前本地工作目录作为基础上下文；该目录下创建、查看和执行的所有任务都归属于这个当前工作目录。
- R3. 第一版每个任务只能选择一个 agent，支持的 agent 仅限 `Claude Code` 与 `Codex CLI`。
- R4. 第一版保留“流程”概念，但只提供一个默认预设流程；任务创建时应显示该流程字段，但该字段为只读不可修改。
- R27. 新建任务默认进入 `初始化` 状态，作为可编辑的草稿态；只有人类显式将其切换到 `待执行` 后，任务才会进入调度队列。
- R22. 任务仅在 `初始化` 与 `重新打开` 状态下允许编辑；进入 `待执行` 后任务定义即锁定，避免排队中的任务与实际执行内容不一致。
- R24. 任务的本地持久化数据必须归属于当前工作目录，不要求跨目录共享任务数据；当人类下次重新打开系统并选择其他工作目录时，应视为进入另一套独立任务空间。
- R26. 系统仅允许在人类刚打开系统时选择当前工作目录；进入该工作目录的任务空间后，本次使用过程中不允许再切换或修改工作目录。如需更换工作目录，必须重新打开系统并重新选择。

**生命周期与执行阶段**
- R5. 任务主状态固定为 `初始化 -> 待执行 -> 执行中 -> 待验证 -> 归档`，并额外支持 `执行失败` 与 `重新打开` 两个显式状态。
- R6. 默认执行流程固定包含三个阶段：`计划 -> 开发 -> 自检`。这三个阶段属于一次执行内部的阶段，不替代任务主状态，且一次执行只启动一个 agent 进程按顺序走完整个流程。
- R7. 当任务从 `初始化` 或 `重新打开` 被人类切换到 `待执行` 后，系统应自动派发所选 agent 按默认流程开始执行。
- R8. 当 agent 执行正常完成时，任务从 `执行中` 进入 `待验证`；当 agent 进程异常退出、中断或系统确认本次执行未完成时，任务自动进入 `执行失败`。
- R9. 当人类在 `待验证` 中判定结果通过时，任务进入 `归档`；当人类判定不通过时，任务进入 `重新打开`，且不会自动重新执行。
- R10. 处于 `重新打开` 的任务，必须由人类手动再次切换到 `待执行` 后，系统才会重新派发。
- R11. 处于 `执行失败` 的任务，不会自动重试；必须由人类手动再次切换到 `待执行` 后，系统才会重新派发。
- R12. 人类可以手动中止处于 `执行中` 的任务；当中止发生后，系统应终止对应 agent 进程，并将任务状态改为 `执行失败`。
- R13. 系统必须支持多个任务并发执行；每个运行中的 agent 进程在同一时刻只处理一个任务，系统通过全局最大并行任务数控制同时执行的任务数量。
- R23. 处于 `执行失败` 的任务不允许直接编辑；人类如需修改任务内容，必须先将其转为 `重新打开`，再进入可编辑状态。
- R21. 当 `待执行` 任务数量超过当前可用并发槽位时，系统必须按先进先出（FIFO）顺序调度任务开始执行。
- R25. 处于 `初始化` 与 `重新打开` 的任务允许修改任务说明与目标 agent，但不允许修改所属工作目录，因为任务始终归属于当前工作目录。
- R28. 人类可以将 `执行失败` 的任务手动转为 `重新打开`，以进入返工编辑态；也可以不编辑，直接手动切回 `待执行` 重新派发。

**观测与验证**
- R14. 看板必须实时展示任务当前主状态、当前执行阶段、当前 attempt 的开始时间与最近事件。
- R15. 看板必须提供实时日志流，至少能连续展示来自 `Claude Code` 与 `Codex CLI` 的标准输出/标准错误输出。
- R16. 系统必须保留任务的执行历史，至少能区分当前任务是否经历过失败、重开与重复执行。
- R17. 验证是人类驱动的显式动作。第一版不要求系统自动判断结果是否通过，也不要求 agent 自主归档。

**产品边界**
- R18. 系统必须同时提供 CLI 与 UI 两种使用入口，但任务模型、状态机、派发、日志采集与持久化能力必须独立于这两层界面。
- R19. UI 第一版以 Electron 桌面应用提供，不要求浏览器 Web 版本。
- R20. 第一版仅面向本地单人使用，不要求多用户协作、权限系统、远程 worker 抢占或云端多租户能力。
- R29. 同一工作目录下，CLI 与 Electron UI 必须指向同一套底层任务数据与实时任务空间；若两者同时接入，应看到一致的任务状态、执行进度与日志结果。
- R30. CLI 第一版主要面向 agent 使用，需支持以命令方式创建任务、切换任务状态、查看任务进度、处理 `重新打开` 与 `归档` 等操作。
- R31. Electron UI 第一版主要面向人类使用，应提供接近传统任务看板的浏览、观察与操作体验。
- R32. CLI 第一版默认以启动命令时所在的 shell 当前工作目录作为任务空间入口，不要求额外显式传入目录参数。

## Success Criteria
- 人类可以在当前工作目录下，从 CLI 或 UI 创建任务，并选择 `Claude Code` 或 `Codex CLI` 发起执行。
- CLI 默认以当前 shell 的工作目录进入对应任务空间，agent 不需要额外传目录参数。
- 人类刚打开系统时可以选择工作目录；进入后不能再切换目录，如需更换目录必须重新打开系统，本次使用只对应这一套独立任务空间。若下次重新打开系统并选择其他目录，应进入另一套独立任务空间。
- 新建任务默认停留在 `初始化`，只有人类显式切到 `待执行` 后才会入队执行。
- 第一版创建任务时会展示默认流程，但该流程值不可编辑。
- agent 可以通过 CLI 在当前工作目录的任务空间内创建任务、派发任务、查看进度，并处理重开或归档等状态操作。
- 人类只能在 `初始化` 与 `重新打开` 状态下修改任务定义；任务进入 `待执行` 后不能再编辑。
- 任务被切换到 `待执行` 后，系统能自动启动对应 agent，并把任务推进到 `执行中`。
- 执行过程中，人类能看到主状态、执行阶段和持续刷新的日志输出。
- agent 正常结束后，任务能进入 `待验证`；agent 异常中断后，任务能自动进入 `执行失败`。
- 人类可以手动中止 `执行中` 任务，且任务会进入 `执行失败`。
- 系统可以并发执行多个任务，但会遵守统一的最大并行任务数限制。
- 当可执行任务多于可用并发槽位时，系统会按 FIFO 顺序启动任务。
- 处于 `执行失败` 的任务不能直接编辑；如需修改，必须先转到 `重新打开`。
- 人类可以把 `待验证` 任务归档，也可以把未通过任务打回 `重新打开`，并在手动切回 `待执行` 后再次派发。
- 人类可以把 `执行失败` 任务手动切回 `待执行` 并重新派发，而系统不会自动重试。
- 同一套核心规则在 CLI 与 Electron UI 中保持一致，不因入口不同而改变任务行为。

## Scope Boundaries
- 第一版不做自定义流程编辑器，不做条件分支、循环、并行节点或通用工作流引擎。
- 第一版不做一个任务绑定多个 agent，也不做多 agent 串行协作。
- 第一版不做多人共享看板、用户权限、审计权限边界或远程执行集群。
- 第一版不做对 `Claude Code` 与 `Codex CLI` 之外 agent 的广泛兼容承诺。
- 第一版不做自动验收、自动回归测试裁决或 AI 审核代替人类验证。
- 第一版不要求人类在任务创建时按 agent 类型单独配置并发上限；并发控制先由系统内部固定。
- 第一版不要求跨工作目录浏览、搜索、聚合或迁移任务数据。
- 第一版不做系统运行中的工作目录切换，也不做一个窗口内管理多个工作目录。

## Key Decisions
- 本地单人优先：先把单机闭环跑通，避免 MVP 被协作、认证与分布式执行复杂度拖垮。
- CLI 与 UI 解耦：把 CLI 和 Electron 都降为界面壳，核心调度能力保持独立，避免后续重复实现。
- CLI 给 agent，GUI 给人：第一版明确区分入口角色，CLI 偏命令式控制与自动化接入，Electron 偏可视化观察与人工操作。
- CLI 以 cwd 进入任务空间：agent 在哪个项目目录里执行命令，就操作该目录下的同一套任务数据。
- 单任务单 agent：优先验证派发与观测闭环，而不是同时解决 agent 编排问题。
- 工作目录是系统上下文：任务空间与本地存储按当前工作目录隔离，避免把目录当成可随意修改的任务字段。
- 单目录会话：用户在进入系统时选择目录，之后整个会话固定在该目录内，避免把第一版做成项目切换器。
- 初始化即草稿：任务创建后先停在草稿态，由人明确提交到 `待执行`，避免刚创建就误入调度队列。
- 固定主状态机：用有限状态约束任务生命周期，降低 UI、CLI 与调度器之间的歧义。
- 预设单流程：第一版只提供一个默认流程，但保留“流程”这个产品概念，为后续扩展留接口。
- 流程字段只读展示：第一版先把流程概念露出来，但不提供修改能力，避免出现假的选择权。
- 单次执行单 agent：`计划 -> 开发 -> 自检` 是一次 execution 内部顺序阶段，由同一个 agent 连续完成，避免阶段切换时的上下文与交接复杂度。
- 全局并发控制：允许同类 agent 多开实例，但先用系统级最大并行任务数兜住资源占用与调度复杂度。
- FIFO 调度：当并发槽位不足时，先保证调度顺序简单透明，不在第一版引入优先级体系。
- 待执行即锁定：任务一旦进入调度队列就冻结定义，避免用户边排队边改内容导致执行语义漂移。
- 执行失败不是返工态：失败结果和返工编辑分离，修改任务前必须先回到 `重新打开`。
- 执行失败有两条人工后续路径：可以直接重新派发，也可以先转为 `重新打开` 再修改。
- 人类负责验收：把“通过/驳回”明确留给人类，避免第一版把验证问题做成另一个 agent 系统。

## Dependencies / Assumptions
- 本机已安装并可从系统环境中调用 `Claude Code` 与 `Codex CLI`。
- 被派发的 agent 能在当前工作目录中以非交互或可被封装的方式启动。
- 本地环境允许系统持续读取 agent 进程输出并判断进程结束状态。

## Outstanding Questions

### Deferred to Planning
- [Affects R8][Technical] 如何为 `Claude Code` 与 `Codex CLI` 定义统一的“异常中断/执行失败”判定规则，同时避免误判正常退出。
- [Affects R12][Technical] 如何安全终止 `Claude Code` 与 `Codex CLI` 的运行进程，并把“人工中止”和“异常失败”区分记录。
- [Affects R13][Technical] 全局最大并行任务数在第一版内部应固定为多少才既能体现并发价值，又不容易压垮本机资源。
- [Affects R14][Technical] 如何在不耦合 CLI 与 Electron 的前提下，把实时状态与日志流稳定广播给两个入口。
- [Affects R16][Technical] 第一版执行历史至少要细到什么粒度，才能既支持重试追踪，又不把本地持久化模型做得过重。
- [Affects R24][Technical] 当前工作目录下的任务数据、日志与元数据应放在什么本地结构里，才能既隔离目录空间，又方便 CLI 与 Electron 共用。
- [Affects R18][Needs research] 核心能力应以本地服务、库加事件总线，还是其他形式暴露给 CLI 与 Electron，哪种更符合后续演进空间。
